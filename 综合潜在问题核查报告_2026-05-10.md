# AI Study Cast 综合潜在问题核查报告

- 我对项目的人工核查
- GLM-5.1 提出的潜在问题

## 已确认的问题

### 1. 启动时手动迁移数据库

位置：
- `backend/app/core/migrations.py`
- `backend/app/main.py`

现状：
- 启动时执行 `ALTER TABLE`
- 旧数据还会被直接 `UPDATE`
- 没有独立迁移版本控制

影响：
- 升级路径不清楚
- 启动失败会影响可用性
- 回滚困难

### 2. CORS 配置写死本地地址

位置：
- `backend/app/main.py`

现状：
- 只允许 `localhost:5173`
- 只允许 `127.0.0.1:5173`

影响：
- 本地可用
- 其他环境需要改代码

### 3. SQLite 外键未显式开启

位置：
- `backend/app/core/database.py`

现状：
- 只看到 `check_same_thread=False`
- 没看到 `PRAGMA foreign_keys=ON`

影响：
- 外键约束和级联删除不稳定

### 4. 前端 Axios 超时过长

位置：
- `frontend/src/api/request.ts`

现状：
- `timeout: 600000`

影响：
- 用户等太久才失败
- 不利于判断网络问题

### 5. DashboardView 单文件过大

位置：
- `frontend/src/views/DashboardView.vue`

现状：
- 约 784 行

影响：
- 维护成本高
- 后续改动容易变乱

### 6. 上传接口缺少明确大小限制

位置：
- `backend/app/api/routes/materials.py`

现状：
- 有上传入口
- 没看到显式文件大小限制

影响：
- 大文件可能拖慢处理
- 容易增加资源压力

### 7. SQLite 并发写入能力弱

位置：
- `backend/app/core/database.py`

现状：
- 使用 SQLite
- 没看到 WAL 或 busy timeout 增强

影响：
- 写并发高时可能锁库

### 8. 时区策略不统一

位置：
- 多处 `datetime.utcnow()`

现状：
- 后端用 UTC 语义
- 前端展示策略没统一说明

影响：
- 显示时间可能和用户预期不一致

### 9. 启动自检不完整

现状：
- 有基础启动逻辑
- 没看到专门的自检清单

影响：
- 缺配置时暴露不够早

### 10. 文件删除函数缺少边界校验

位置：
- `backend/app/services/audio_service.py`

现状：
- 传入路径后直接处理
- 绝对路径会直接尝试删除

影响：
- 函数本身边界不够硬

### 11. AI 连接测试错误信息可能泄露细节

位置：
- `backend/app/api/routes/settings.py`

现状：
- 会把 `str(e)` 返回前端
- 也会把 `resp.text` 写入日志

影响：
- 可能暴露外部服务返回细节

### 12. `.env.example` 与默认超时不一致

位置：
- `.env.example`
- `backend/app/core/config.py`

现状：
- 示例值是 `60`
- 代码默认值是 `120`

影响：
- 配置预期不一致

### 13. 路由里的重复 JSON 辅助函数

位置：
- `backend/app/api/routes/ai.py`
- `backend/app/api/routes/knowledge_points.py`
- `backend/app/api/routes/materials.py`
- `backend/app/api/routes/flashcards.py`

现状：
- `_dump_json` / `_load_json` 多处重复

影响：
- 不是功能 bug
- 但维护会越来越散

## 部分成立

### 1. AI JSON 解析不够健壮

结论：
- 方向对
- 但不能说“没兜底”

原因：
- `json_parser.py` 已支持纯 JSON、代码块、文本截取
- `ai_service.py` 还有修复流程

更准确的说法：
- 现在是“有兜底，但还不够强”

### 2. AI 成功日志写入位置不够清晰

结论：
- 风险存在
- 但不是明确的“双重成功日志” bug

更准确的说法：
- `chat()` 的 `finally` 里补写成功日志，可读性差
- 维护时容易误判

### 3. 复习逻辑的事务一致性值得关注

结论：
- 原报告说法偏满

现状：
- `review_service.py` 本身没看到它描述的“双表同时写”
- 但答题、错题、复习记录这类逻辑本来就应该重点关注事务

### 4. `/upload-text` 会留下未入库文件

结论：
- 方向对
- 但不是“孤立数据库记录”

现状：
- 这里只是把文本文件存到 `data/uploads`
- 如果后续不导入，文件会留下

### 5. 多选答案的边界处理有问题

结论：
- 基本成立

现状：
- 英文逗号分隔正常
- 中文逗号等变体容易判错

### 6. 前端错误处理不统一

结论：
- 基本成立

现状：
- 请求层只做了 `Promise.reject(error)`
- 没有统一中文错误提取

### 7. `@app.on_event("startup")` 的表述要更谨慎

结论：
- 仍然是需要关注的点
- 但更准确的说法是“FastAPI 更推荐 lifespan”

现状：
- 当前代码仍在使用 `startup`
