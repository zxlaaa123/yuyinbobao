## 已修复问题

### 1. CORS 配置写死本地地址

涉及文件：

- `backend/app/core/config.py`
- `backend/app/main.py`
- `.env.example`

修复方法：

- 新增 `get_cors_origins()`。
- 优先读取 `CORS_ORIGINS`。
- 兼容 `FRONTEND_URL`。
- 保留本地默认地址：
  - `http://localhost:5173`
  - `http://127.0.0.1:5173`
- `main.py` 中 `allow_origins` 改为调用配置函数。

效果：

- 后端 CORS 不再只能写死在代码里。
- 后续换前端地址只需要改 `.env`。

### 2. SQLite 外键未显式开启

涉及文件：

- `backend/app/core/database.py`

修复方法：

- 为 SQLAlchemy engine 增加 `connect` 事件。
- 每次 SQLite 连接创建时执行：
  - `PRAGMA foreign_keys=ON`
  - `PRAGMA journal_mode=WAL`
  - `PRAGMA busy_timeout=30000`
- SQLite 连接增加 `timeout: 30`。

效果：

- SQLite 外键约束生效。
- 并发写入等待能力增强。
- WAL 模式减少读写互相阻塞。

### 3. 前端 Axios 超时过长

涉及文件：

- `frontend/src/api/request.ts`
- `.env.example`

修复方法：

- 原固定 `timeout: 600000` 改为读取 `VITE_API_TIMEOUT`。
- 默认值改为 `120000` 毫秒。
- `.env.example` 新增 `VITE_API_TIMEOUT=120000`。

效果：

- 前端请求超时可配置。
- 避免默认等待 10 分钟导致用户误以为页面卡死。

### 4. 上传接口缺少明确大小限制

涉及文件：

- `backend/app/core/config.py`
- `backend/app/api/routes/materials.py`
- `.env.example`

修复方法：

- 新增 `MAX_UPLOAD_SIZE_MB`。
- 新增 `MAX_UPLOAD_SIZE_BYTES`。
- `.env.example` 默认：
  - `MAX_UPLOAD_SIZE_MB=10`
- 上传接口读取文件时使用：
  - `file.read(MAX_UPLOAD_SIZE_BYTES + 1)`
- 超过限制返回：
  - HTTP 413
  - 中文错误提示

效果：

- 避免超大文件一次性读入内存。
- 上传限制可配置。

### 5. AI / TTS 错误信息可能泄露细节

涉及文件：

- `backend/app/api/routes/ai.py`
- `backend/app/api/routes/materials.py`
- `backend/app/api/routes/tts.py`
- `backend/app/api/routes/settings.py`
- `backend/app/api/routes/flashcards.py`

修复方法：

- 对用户可见错误做脱敏。
- 不再直接返回 `str(e)`、底层 HTTP 异常、SDK 异常内容。
- AI 失败统一返回类似：
  - `AI 调用失败，请检查模型配置或稍后重试`
- TTS 失败统一返回类似：
  - `TTS 生成失败，请检查 TTS 配置或稍后重试`
- 设置页连接测试仍保留内部日志错误信息，便于排查。

效果：

- 减少外部接口错误、URL、网络细节暴露给前端用户。
- 保留后端排查能力。

### 6. 文件删除函数缺少目录边界校验

涉及文件：

- `backend/app/services/audio_service.py`

修复方法：

- 删除音频文件前解析真实路径。
- 计算 `AUDIO_DIR.resolve()`。
- 使用 `path.is_relative_to(audio_root)` 判断目标是否位于音频目录内。
- 不在音频目录内则直接返回 `False`。

效果：

- 避免传入异常绝对路径时删除 `data/audio` 之外的文件。
- 删除行为限制在音频目录范围内。

### 7. `.env.example` 与默认超时不一致

涉及文件：

- `.env.example`
- `backend/app/core/config.py`

修复方法：

- `.env.example` 中 `AI_TIMEOUT` 从 `60` 改为 `120`。
- 与后端默认值保持一致。

效果：

- 示例配置和代码默认值一致。
- 降低配置误解。

### 8. 路由里的重复 JSON 辅助函数

涉及文件：

- `backend/app/api/routes/ai.py`
- `backend/app/api/routes/materials.py`
- `backend/app/api/routes/knowledge_points.py`
- `backend/app/api/routes/flashcards.py`
- `backend/app/utils/json_helpers.py`

修复方法：

- 复用已有公共工具：
  - `dump_json`
  - `load_json`
- 删除路由文件内重复的 `_dump_json`、`_load_json` 实现。
- 保留别名导入，减少调用处改动。

效果：

- JSON 序列化/反序列化逻辑集中。
- 后续维护只改公共 helper。

## 中等风险修复补充

本阶段继续修复中等风险项。仍采用最小修复策略，不做大规模重构。

### 9. 多选答案边界处理

涉及文件：

- `backend/app/api/routes/practice.py`

修复方法：

- 新增 `_answer_to_text()`，统一处理字符串和数组答案。
- 多选答案支持：
  - 数组
  - 英文逗号
  - 中文逗号
  - 分号
  - 顿号
  - 空白分隔
- 新增 `_normalize_multi_answer()`：
  - 转大写
  - 去空值
  - 去重
  - 排序
- 新增 `_load_options()` 读取题目选项。
- 新增 `_validate_answer()`：
  - 空答案返回中文错误
  - 多选答案必须在题目选项范围内

效果：

- `["A", "C"]`、`A,C`、`A，C` 等输入可统一判断。
- 防止提交不存在的选项。
- 多选判题更稳定。

### 10. 答题提交与复习逻辑事务一致性

涉及文件：

- `backend/app/api/routes/practice.py`
- `backend/app/api/routes/practice_sessions.py`

修复方法：

- `submit_answer()` 中将以下操作纳入同一事务：
  - 写入答题记录
  - 写入或更新错题记录
  - 更新知识点复习状态
- 任一步失败则 `db.rollback()`。
- 返回中文错误：
  - `提交答案失败，请稍后重试`
- `create_practice_session()` 中将练习会话和练习明细写入纳入同一事务。
- 任一步失败则 `db.rollback()`。
- 返回中文错误：
  - `保存练习记录失败，请稍后重试`

效果：

- 避免答题记录已保存但复习状态未更新。
- 避免练习会话已保存但明细缺失。
- 数据一致性更好。

### 11. `/upload-text` 会留下未入库文件

涉及文件：

- `backend/app/api/routes/materials.py`
- `frontend/src/api/material.ts`

修复方法：

- 上传预览接口不再写入 `data/uploads/`。
- 接口只负责：
  - 校验文件类型
  - 校验文件大小
  - 解析文本编码
  - 返回标题、正文、来源文件名
- 返回字段 `saved_path` 改为 `null`。
- 前端类型同步改为：
  - `saved_path: string | null`

效果：

- 用户只上传预览、不保存资料时，不再留下孤立文件。
- 导入流程更干净。

### 12. 启动自检增强

涉及文件：

- `backend/app/core/startup.py`
- `backend/app/main.py`

修复方法：

- 新增 `ensure_runtime_environment(engine)`。
- 启动时检查并创建运行目录：
  - `data/`
  - `data/audio/`
  - `data/uploads/`
  - `data/vector_store/`
  - `data/backups/`
- 对每个目录执行写入探针：
  - 创建 `.write_test`
  - 删除 `.write_test`
- 检查数据库连通性：
  - `SELECT 1`
- `main.py` 的 startup 中先执行运行环境检查，再执行建表和运行时列迁移。

效果：

- 启动时能更早发现目录权限、路径、数据库连接问题。
- 比只创建目录更完整。

### 13. 前端错误处理不统一

涉及文件：

- `frontend/src/api/request.ts`
- `frontend/src/utils/error.ts`

修复方法：

- 复用已有 `getErrorMessage()`。
- Axios 响应拦截器中统一转换错误：
  - `Promise.reject(new Error(getErrorMessage(error)))`
- 后端连接失败、服务端中文错误、网络错误统一走同一解析逻辑。

效果：

- 页面层收到的错误更稳定。
- 减少 `undefined`、`Network Error`、原始 Axios 错误直接显示给用户。
